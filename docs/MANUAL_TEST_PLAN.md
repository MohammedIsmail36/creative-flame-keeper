# خطة الاختبار اليدوي الشاملة - من منظور المستخدم النهائي

> **الإصدار**: 2.2 (محدّث ليشمل: نظام الضريبة المركزي، تقارير دوران المخزون السبعة، شريط التنقل المُعرَّب، مساري "بدون ضريبة" و"بضريبة"، **اختبار الخصومات (سطر/إجمالي)**، **التحقق من `net_total`**، **شاشات كشف الحساب الجديدة + RPC**، **اختبارات WAC الموسّعة** (خصم الشراء، مرتجع بسعر مختلف، تسوية المخزون))

## الهدف
اختبار النظام المحاسبي **مرتين**:
1. **المسار A — بدون ضريبة**: للتأكد من أن النظام يعمل بشكل سليم عند تعطيل الضريبة (لا تظهر أعمدة/قيود ضريبة).
2. **المسار B — بضريبة 15%**: للتأكد من أن الضريبة تُحتسب وتُرحّل على الحسابات الصحيحة.

## الافتراضات العامة
- **العملة**: الجنيه المصري (EGP) أو ريال (SAR) حسب الإعداد
- **بداية السنة المالية**: 01/01 من السنة الحالية
- **نسبة الضريبة (للمسار B)**: 15%
- **قاعدة بيانات**: نظيفة (يُفضّل تشغيل `seed-system` قبل البدء)

   - [ ] **متوقع**: ❌ رفض مع رسالة "حساب ضريبة المشتريات إلزامي"

### 3.6 🆕 اختبار الخصومات (المسار A — بدون ضريبة)

> ⚠️ **مستحدث** — يختبر الخصم على مستوى السطر والخصم الإجمالي مع التأكد من `net_total` والقيود.

#### 3.6.1 خصم على مستوى السطر فقط
**فاتورة بيع جديدة**: 10 وحدات × 100 = 1,000، **خصم السطر = 50**

| البند | القيمة المتوقعة |
|------|----------------|
| إجمالي السطر (`total`) | 1,000.00 |
| صافي السطر (`net_total`) | **950.00** |
| إجمالي الفاتورة | 950.00 |
| COGS | 10 × 55 = **550.00** |

| الحساب | مدين | دائن |
|--------|------|------|
| العملاء (1103) | 950 | - |
| إيرادات المبيعات (4101) | - | 950 |
| ت.ب.م (5101) | 550 | - |
| المخزون (1104) | - | 550 |

- [ ] القيد يستخدم **صافي البيع (950)** وليس الإجمالي (1000)
- [ ] حقل `net_total` في `sales_invoice_items` = 950

#### 3.6.2 خصم إجمالي على مستوى الفاتورة فقط
**فاتورة بيع جديدة**: 10 وحدات × 100 = 1,000، **بدون خصم سطر**، **خصم إجمالي = 100**

| البند | القيمة المتوقعة |
|------|----------------|
| الإجمالي الفرعي | 1,000.00 |
| الخصم الإجمالي | 100.00 |
| **الإجمالي النهائي** | **900.00** |
| COGS | 550.00 |

| الحساب | مدين | دائن |
|--------|------|------|
| العملاء (1103) | 900 | - |
| إيرادات المبيعات (4101) | - | 900 |
| ت.ب.م (5101) | 550 | - |
| المخزون (1104) | - | 550 |

- [ ] رصيد العميل يزيد بـ 900 فقط
- [ ] القيد يحترم الخصم الإجمالي

#### 3.6.3 منع الجمع بين خصم السطر والخصم الإجمالي (EC ضمن الفاتورة)
- في فاتورة فيها خصم سطر، حاول إدخال خصم إجمالي
- [ ] **متوقع**: ❌ النظام يمنع الجمع (تعطيل الحقل أو رسالة خطأ)

#### 3.6.4 مرتجع لفاتورة مخصومة
- مرتجع 5 وحدات من فاتورة 3.6.1 (التي لها خصم سطر = 50 على 10 وحدات)
- نسبة الخصم على الوحدة = 50/10 = 5
- **القيمة المتوقعة للمرتجع**: 5 × (100 - 5) = **475.00**

#### 3.6.5 🆕 أثر خصم فاتورة الشراء الإجمالي على WAC

**شرط مسبق**: المنتج له رصيد سابق (مثلاً: كمية 100، قيمة 5,500 → WAC = 55.00).

**فاتورة شراء جديدة**: 100 وحدة × 60 = 6,000، **خصم إجمالي = 500** → `net_total = 5,500`.

| البند | الحساب | القيمة |
|-------|--------|--------|
| الإجمالي قبل الخصم | 100 × 60 | 6,000.00 |
| الخصم الإجمالي | — | 500.00 |
| **net_total (يُسجَّل في المخزون)** | — | **5,500.00** |
| WAC الجديد | (5,500 + 5,500) / (100 + 100) | **55.00** |

> 💡 **القاعدة**: WAC يستخدم `net_total / quantity` (وليس `total / quantity`)، طبقاً لـ `mem://features/purchase-invoice-discount-logic`.

- [ ] قيمة المخزون المُضافة = **5,500.00** (وليس 6,000)
- [ ] WAC المحسوب يستخدم `net_total` وليس الإجمالي قبل الخصم
- [ ] أي بيع لاحق يستخدم WAC المنخفض في حساب COGS
- [ ] القيد: مدين المخزون 5,500 / دائن الموردون 5,500

- [ ] قيد المرتجع يستخدم 475 (وليس 500)
- [ ] رصيد العميل ينقص 475
