

# خطة اختبار شاملة للنظام المحاسبي — سيناريو محل قائم

## السيناريو
محل تجاري قائم يبدأ استخدام النظام. لديه منتجات موجودة، عملاء وموردين بأرصدة سابقة.

## بيانات الاختبار المرجعية

```text
المنتجات:
  - لابتوب HP     | سعر شراء: 8,000 | سعر بيع: 10,000 | كمية افتتاحية: 5
  - ماوس لاسلكي   | سعر شراء: 100   | سعر بيع: 200    | كمية افتتاحية: 20
  - شاشة سامسونج  | سعر شراء: 3,000 | سعر بيع: 4,500  | كمية افتتاحية: 3

العملاء:
  - عميل أحمد | رصيد افتتاحي: 5,000 (مستحق عليه)
  - عميل سارة | رصيد افتتاحي: 0

الموردين:
  - مورد التقنية | رصيد افتتاحي: 3,000 (مستحق له)
  - مورد الشاشات | رصيد افتتاحي: 0
```

## مراحل الاختبار (بالترتيب)

### المرحلة 0: تهيئة النظام
**الإجراء:** تشغيل seed-system من صفحة إعداد النظام
**التحقق:**
- تسجيل دخول بـ admin@system.com / admin123456
- شجرة الحسابات (29 حساب) موجودة
- إعدادات الشركة منشأة
- ميزان المراجعة = 0 مدين و 0 دائن (فارغ)

### المرحلة 1: إعدادات الشركة
**الإجراء:** الإعدادات → تعديل اسم الشركة، العملة، نسبة الضريبة (14%)، بادئات الفواتير
**التحقق:** الإعدادات محفوظة وتظهر في الـ Sidebar

### المرحلة 2: البيانات الأساسية (Lookups)
**الإجراء:** إنشاء وحدات (قطعة)، ماركات (HP، سامسونج)، تصنيفات (إلكترونيات، ملحقات)
**التحقق:** تظهر كخيارات في نموذج المنتج

### المرحلة 3: المنتجات + الأرصدة الافتتاحية
**الإجراء:** إنشاء 3 منتجات بأسعار وكميات افتتاحية
**التحقق:**
- كمية كل منتج صحيحة في صفحة المنتجات
- حركات المخزون: 3 حركات نوع `opening_balance`
- ميزان المراجعة: حساب المخزون (1104) مدين = (5×8000 + 20×100 + 3×3000) = 51,000
- حساب رأس المال (3101) دائن = 51,000
- **الميزان متوازن: مدين 51,000 = دائن 51,000**

### المرحلة 4: العملاء + الأرصدة الافتتاحية
**الإجراء:** إنشاء عميل أحمد (رصيد 5,000) + عميل سارة (رصيد 0)
**التحقق:**
- رصيد أحمد = 5,000 في صفحة العملاء
- قيد افتتاحي: مدين العملاء 1103 = 5,000 / دائن رأس المال 3101 = 5,000
- ميزان المراجعة: مدين 56,000 = دائن 56,000

### المرحلة 5: الموردين + الأرصدة الافتتاحية
**الإجراء:** إنشاء مورد التقنية (رصيد 3,000) + مورد الشاشات (رصيد 0)
**التحقق:**
- رصيد مورد التقنية = 3,000
- قيد: مدين رأس المال 3101 = 3,000 / دائن الموردين 2101 = 3,000
- ميزان المراجعة: مدين 56,000 = دائن 56,000 (لأن المدين نقص 3000 من رأس المال والدائن زاد 3000 في الموردين ونقص 3000 من رأس المال) → **مدين 56,000 = دائن 56,000**

### المرحلة 6: فاتورة شراء (بدون خصم)
**الإجراء:** فاتورة شراء من مورد التقنية:
- 10 لابتوب × 8,500 = 85,000
- حفظ كمسودة ثم اعتماد
**التحقق:**
- كمية اللابتوب = 5 + 10 = 15
- رصيد مورد التقنية = 3,000 + 85,000 = 88,000
- حركة مخزون: unit_cost = 8,500
- ميزان المراجعة متوازن

### المرحلة 7: فاتورة شراء (مع خصم فاتورة)
**الإجراء:** فاتورة شراء من مورد الشاشات:
- 5 شاشة × 3,200 = 16,000
- خصم فاتورة = 1,000
- الإجمالي = 15,000
**التحقق:**
- net_total لكل سطر = 16,000 × (1 - 1000/16000) = 15,000
- حركة مخزون: unit_cost = 15,000 / 5 = 3,000
- كمية الشاشات = 3 + 5 = 8
- رصيد مورد الشاشات = 15,000
- ميزان المراجعة متوازن

### المرحلة 8: فاتورة مبيعات (بدون خصم)
**الإجراء:** فاتورة مبيعات لعميل سارة:
- 2 لابتوب × 10,000 = 20,000
- بدون ضريبة، حفظ واعتماد
**التحقق:**
- كمية اللابتوب = 15 - 2 = 13
- رصيد سارة = 20,000
- COGS = متوسط سعر الشراء × 2 (من inventory_movements)
- ميزان المراجعة متوازن

### المرحلة 9: فاتورة مبيعات (مع خصم فاتورة)
**الإجراء:** فاتورة مبيعات لعميل أحمد:
- 1 لابتوب × 10,000 = 10,000
- 5 ماوس × 200 = 1,000
- خصم فاتورة = 500
- الإجمالي = 10,500
**التحقق:**
- net_total اللابتوب = 10,000 × (1 - 500/11000) ≈ 9,545.45
- net_total الماوس = 1,000 × (1 - 500/11000) ≈ 954.55
- رصيد أحمد = 5,000 + 10,500 = 15,500
- كمية اللابتوب = 12، ماوس = 15
- ميزان المراجعة متوازن

### المرحلة 10: فاتورة مبيعات (مع خصم سطر)
**الإجراء:** فاتورة مبيعات لعميل سارة:
- 2 شاشة × 4,500 - خصم 200 = 8,800
- اعتماد
**التحقق:**
- net_total = total = 8,800 (الخصم مطبق بالفعل)
- رصيد سارة = 20,000 + 8,800 = 28,800
- كمية الشاشات = 8 - 2 = 6
- ميزان المراجعة متوازن
- **تأكيد القفل المتبادل**: عدم إمكانية إدخال خصم فاتورة مع وجود خصم سطر

### المرحلة 11: مرتجع مبيعات
**الإجراء:** مرتجع من عميل سارة:
- 1 لابتوب × 10,000 (نفس سعر البيع الأصلي)
- ربط بالفاتورة الأصلية، اعتماد
**التحقق:**
- كمية اللابتوب = 12 + 1 = 13
- رصيد سارة = 28,800 - 10,000 = 18,800
- ميزان المراجعة متوازن

### المرحلة 12: مرتجع مشتريات
**الإجراء:** مرتجع لمورد التقنية:
- 2 لابتوب × 8,500
- اعتماد
**التحقق:**
- كمية اللابتوب = 13 - 2 = 11
- رصيد مورد التقنية = 88,000 - 17,000 = 71,000
- ميزان المراجعة متوازن

### المرحلة 13: سند قبض (دفعة عميل)
**الإجراء:** دفعة من عميل أحمد = 10,000 نقداً
**التحقق:**
- رصيد أحمد = 15,500 - 10,000 = 5,500
- حساب الصندوق (1101) مدين += 10,000
- ميزان المراجعة متوازن

### المرحلة 14: سند صرف (دفعة مورد)
**الإجراء:** دفعة لمورد التقنية = 20,000 نقداً
**التحقق:**
- رصيد مورد التقنية = 71,000 - 20,000 = 51,000
- حساب الصندوق (1101) دائن += 20,000
- ميزان المراجعة متوازن

### المرحلة 15: مصروفات
**الإجراء:** إنشاء نوع مصروف (إيجار) → تسجيل مصروف 2,000 نقداً → اعتماد
**التحقق:**
- حساب الإيجار مدين += 2,000
- حساب الصندوق دائن += 2,000
- ميزان المراجعة متوازن

### المرحلة 16: قيد يدوي
**الإجراء:** قيد يومية يدوي (مثلاً تحويل 5,000 من الصندوق للبنك)
**التحقق:**
- الصندوق دائن 5,000 / البنك مدين 5,000
- ميزان المراجعة متوازن

## مرحلة التقارير — التحقق الشامل

### المرحلة 17: التقارير المالية
| التقرير | ماذا نتحقق |
|---|---|
| ميزان المراجعة | مدين = دائن (متوازن) |
| قائمة الدخل | إيرادات = مجموع صافي المبيعات، مصروفات تشمل COGS + إيجار |
| الميزانية العمومية | أصول = خصوم + حقوق ملكية |
| الأرباح والخسائر | هامش ربح = (صافي مبيعات - COGS) / صافي مبيعات |

### المرحلة 18: تقارير تشغيلية
| التقرير | ماذا نتحقق |
|---|---|
| تقرير المبيعات | إجمالي = مجموع الفواتير المعتمدة (بعد الخصم) |
| تقرير المشتريات | إجمالي = مجموع فواتير الشراء المعتمدة |
| تقرير المخزون | كميات وقيم تطابق الحركات |
| أعمار الديون | رصيد أحمد 5,500 وسارة 18,800 |
| كشف حساب عميل | تفاصيل حركات أحمد وسارة |
| كشف حساب مورد | تفاصيل حركات الموردين |

### المرحلة 19: تقارير تحليلية
| التقرير | ماذا نتحقق |
|---|---|
| تحليل المنتجات | إيرادات كل منتج تستخدم net_total (بعد الخصم) |
| تحليلات النمو | هامش ربح صحيح يعتمد على صافي المبيعات |
| أرصدة الحسابات | كل حساب رصيده يطابق ميزان المراجعة |
| دوران المخزون | نسبة دوران = COGS / متوسط المخزون |

### المرحلة 20: لوحة التحكم (Dashboard)
| البيان | القيمة المتوقعة |
|---|---|
| إجمالي المبيعات | مجموع grandTotal للفواتير المعتمدة |
| إجمالي المشتريات | مجموع grandTotal لفواتير الشراء |
| المنتجات الأكثر مبيعاً | مرتبة حسب net_total |
| هامش الربح | صحيح بناءً على صافي المبيعات |
| تنبيهات المخزون المنخفض | تظهر إذا كانت الكمية < min_stock_level |

## آلية التنفيذ

سأقوم بالاختبار برمجياً عبر استعلامات قاعدة البيانات بعد كل مرحلة للتحقق من:
1. أرصدة الحسابات في journal_entry_lines
2. أرصدة العملاء/الموردين
3. كميات المنتجات وحركات المخزون
4. توازن ميزان المراجعة

عند اكتشاف خطأ: نحدد السبب، نصلح الكود، ثم نعيد من أقرب نقطة سليمة (أو من الصفر إذا لزم الأمر عبر أداة تصفير النظام).

## الملفات المتأثرة (محتملة)
أي ملف يظهر فيه خطأ أثناء الاختبار سيتم إصلاحه فوراً. التركيز على:
- `SalesInvoiceForm.tsx` / `PurchaseInvoiceForm.tsx` — منطق الحفظ والاعتماد
- `Dashboard.tsx` — KPIs وتجميع البيانات
- ملفات التقارير في `src/pages/reports/`
- `entity-balance.ts` — حساب أرصدة العملاء والموردين
