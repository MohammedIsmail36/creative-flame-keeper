

## السماح بحفظ الفواتير كمسودة بدون مورد/عميل + تحديث القائمة بدون إعادة تحميل

### المشكلة
عند إدخال فاتورة بـ 50 سطر ثم اكتشاف أن المورد غير مُسجَّل، يضطر المستخدم لمغادرة الشاشة لإضافته → تُفقد كل البيانات. كما أن المسودة تشترط حالياً اختيار المورد/العميل.

### الحل (وفق أفضل الممارسات — مماثل لـ Odoo / Zoho)

نطبّق ثلاث طبقات تكاملية:

#### 1) تخفيف شروط حفظ المسودة
- **المسودة (Draft)**: لا يُشترط المورد/العميل ولا الأصناف. فقط أي بيانات يدخلها المستخدم تُحفظ.
- **الترحيل (Post / إصدار الفاتورة)**: تبقى كل التحققات صارمة كما هي (مورد + أصناف + كميات + أسعار).
- إزالة النجمة الحمراء `*` من حقل المورد/العميل عندما تكون الحالة مسودة، وإظهار تلميح: «اختياري للمسودة — مطلوب عند الإصدار».

#### 2) إضافة مورد/عميل **inline** من داخل الفاتورة (بدون مغادرة الصفحة)
- إضافة زر «➕ مورد جديد» / «➕ عميل جديد» داخل `LookupCombobox` (في أسفل قائمة النتائج، أو بجانب الحقل).
- عند الضغط: يفتح **Dialog صغير** يحتوي الحقول الأساسية فقط (الاسم + الهاتف + الكود يُولَّد تلقائياً).
- عند الحفظ: يُضاف المورد للقاعدة، يُحدَّث `suppliers[]` محلياً، ويُختار تلقائياً في الحقل.
- النتيجة: لا إعادة تحميل، لا فقدان للبنود الـ 50.

#### 3) Auto-save للمسودة (شبكة أمان)
- حفظ تلقائي صامت كل 30 ثانية إذا كانت هناك تعديلات (`isDirty`) وحالة المسودة، بشرط وجود **بند واحد على الأقل**.
- مؤشر صغير «✓ تم الحفظ تلقائياً HH:MM» بجانب زر الحفظ.
- يحمي من إغلاق التبويب فجأة أو انقطاع الإنترنت.

### السلوك النهائي للمستخدم
1. يفتح فاتورة شراء، يبدأ بإدخال 50 بند.
2. يكتشف أن المورد غير موجود → يضغط «➕ مورد جديد» داخل حقل المورد → يكتب الاسم → حفظ.
3. يُختار المورد فوراً، تبقى البنود سليمة.
4. لو أراد التوقف الآن: «حفظ كمسودة» يعمل حتى لو لم يختر المورد أصلاً.
5. عند «إصدار الفاتورة»: التحقق الكامل يضمن سلامة البيانات المحاسبية.

### ملاحظة محاسبية
لا يوجد أي تأثير على القيود أو المخزون: المسودة لا تُولّد JV ولا حركة مخزون. التحقق الصارم يبقى مرتبطاً بالترحيل فقط.
