

# خطة مراجعة شاملة لجاهزية النظام للإنتاج

## الهدف
إنتاج ملف توثيقي شامل (`docs/PRODUCTION_READINESS_AUDIT.md`) يرصد جميع المشكلات الحدية في النظام مرتبة حسب الأولوية والفئة، ليكون خارطة طريق واضحة للوصول لمرحلة النشر الإنتاجي.

---

## المنهجية: 3 محاور للمراجعة

### المحور الأول — المنطق التجاري والمحاسبي (الأولوية القصوى)

سأفحص بعمق:

1. **سلامة القيود المحاسبية**
   - توازن القيود (Debit = Credit) في كل سيناريو
   - دقة COGS بطريقة WAC في حالات: مخزون صفري، مرتجعات قبل بيع، تسويات
   - حسابات الضريبة مع الخصومات (سطر/إجمالي)
   - معالجة الكسور العشرية (round2 + Number.EPSILON)

2. **الفواتير والمرتجعات**
   - منع تجاوز كمية المرتجع للأصلي
   - منع تكرار التسويات
   - حماية الفترات المغلقة (locked_until)
   - حدود أيام الاسترجاع (return_days_limit)

3. **المخزون**
   - منع البيع بكمية أكبر من المتاح (stock_enforcement)
   - تطابق GL مع inventory_movements (reconciliation)
   - معالجة الأرصدة الافتتاحية وقيود 1104/3101

4. **المدفوعات والتسويات**
   - رفض دفعة > المتبقي
   - منع تخصيص سالب أو صفري
   - دقة paid_amount في الفواتير

5. **الإقفال السنوي**
   - دقة قيد الإقفال (3102)
   - توازن الميزانية بعد الإقفال
   - عكسية العملية

6. **الصلاحيات والأمان**
   - تحقق RLS لكل جدول
   - فصل الأدوار (admin/accountant/sales)
   - حماية has_role من التلاعب

---

### المحور الثاني — تجربة المستخدم (UX)

سأفحص:

1. **التحقق من المدخلات (Validation)**
   - رسائل خطأ واضحة بالعربية
   - منع الحفظ مع بيانات ناقصة
   - تحقق Zod في كل النماذج الحرجة

2. **حالات التحميل والأخطاء**
   - Skeletons أثناء التحميل
   - ErrorBoundary لكل صفحة رئيسية
   - رسائل toast واضحة عند الفشل

3. **التنقل والـ Breadcrumb**
   - تطابق المسارات مع routeLabels
   - حفظ الحالة عند الـ refresh
   - تحذير قبل المغادرة (use-before-unload)

4. **النماذج الطويلة**
   - الحفظ كمسودة
   - الإدخال السريع (Tab/Enter)
   - تحذير عند فقدان التغييرات

5. **الطباعة والتصدير**
   - PDF بالعربية (RTL + Arabic fonts)
   - Excel/CSV بترميز صحيح
   - معاينة قبل الطباعة

6. **الواجهة الموبايل**
   - استجابة الجداول للشاشات الصغيرة
   - السايدبار على الموبايل

---

### المحور الثالث — الأداء (Performance)

سأفحص:

1. **استعلامات قاعدة البيانات**
   - حد الـ 1000 صف من Supabase
   - استخدام pagination في الجداول الكبيرة
   - Indexes المفقودة على الأعمدة المُستعلمة

2. **استعلامات React Query**
   - Cache invalidation الصحيح
   - staleTime مناسب لكل نوع بيانات
   - منع استعلامات N+1

3. **حجم الـ Bundle**
   - Lazy loading للصفحات
   - Code splitting
   - Tree shaking

4. **التقارير الثقيلة**
   - تحليل Inventory Turnover
   - Profit & Loss
   - Growth Analytics
   - استعلامات متوازية vs متسلسلة

5. **Realtime subscriptions**
   - تنظيف الاشتراكات عند unmount
   - منع تسرب الذاكرة

## مراحل التنفيذ

### المرحلة 1 — جمع البيانات (Discovery)
- فحص جميع ملفات `src/pages/` للنماذج
- فحص `src/lib/` للمنطق المحاسبي
- فحص `src/hooks/` لأنماط الاستعلامات
- فحص `supabase/migrations/` لـ RLS و triggers
- فحص `supabase/functions/` لـ edge functions
- مراجعة ملفات الاختبارات الموجودة (ما هو مغطى؟ ما هو مفقود؟)

### المرحلة 2 — التحليل والتصنيف
لكل مشكلة يتم رصدها:
- **الفئة**: منطق / UX / أداء / أمان
- **الخطورة**: حرجة (Blocker) / عالية / متوسطة / منخفضة
- **الموقع**: الملف والسطر
- **الوصف**: ما المشكلة بالضبط
- **التأثير**: ماذا يحدث في الإنتاج
- **الحل المقترح**: خطوات الإصلاح
- **التحقق**: كيف نتأكد من الإصلاح

### المرحلة 3 — إنتاج الملف التوثيقي

سيتم إنشاء `docs/PRODUCTION_READINESS_AUDIT.md` بهذا الهيكل:

```text
1. ملخص تنفيذي
   - عدد المشكلات حسب الفئة والخطورة
   - تقدير الجهد الكلي
   - قائمة الـ Blockers

2. المنطق المحاسبي والتجاري
   2.1 القيود المحاسبية
   2.2 الفواتير والمرتجعات
   2.3 المخزون
   2.4 المدفوعات
   2.5 الإقفال السنوي
   2.6 الصلاحيات

3. تجربة المستخدم
   3.1 التحقق من المدخلات
   3.2 حالات التحميل والأخطاء
   3.3 النماذج
   3.4 الطباعة والتصدير
   3.5 الموبايل

4. الأداء
   4.1 قاعدة البيانات
   4.2 React Query
   4.3 Bundle Size
   4.4 التقارير الثقيلة

5. الأمان
   5.1 RLS Policies
   5.2 Edge Functions
   5.3 Secrets

6. خارطة الطريق المقترحة
   - Sprint 1: Blockers
   - Sprint 2: High priority
   - Sprint 3: Polish

7. قائمة فحص ما قبل النشر (Pre-launch Checklist)
```

كل مشكلة بالشكل:
```text
### [LOGIC-001] عنوان المشكلة
- الخطورة: 🔴 حرجة
- الموقع: src/pages/SalesInvoiceForm.tsx:245
- الوصف: ...
- التأثير: ...
- الحل: ...
- التحقق: ...
```

## ملاحظات مهمة

- **لن أُجري أي إصلاحات في هذه المرحلة** — الهدف فقط رصد وتوثيق
- التركيز على المشكلات الحقيقية الموجودة في الكود، لا الافتراضية
- كل مشكلة مرفقة بمرجع للملف والسطر للتحقق
- الخطة تستهدف نظاماً محاسبياً للإنتاج الفعلي، فالمنطق المحاسبي له الأولوية القصوى
- بعد موافقتك سأبدأ مرحلة الفحص العميق وإنتاج الملف
