

## خطة إنشاء ملفي الجاهزية الإنتاجية والاختبار اليدوي

### الهدف
إنتاج **ملفين جديدين منفصلين** يمثلان "مصدر الحقيقة" النهائي للنظام بعد المراجعة الشاملة، ليكونا أساس قرار الانتقال للإنتاج.

---

## الملف الأول — `docs/PRODUCTION_READINESS_V3.md`

تقرير جاهزية إنتاجية احترافي مبني على فحص فعلي للكود الحالي (وليس افتراضات قديمة).

### هيكل الملف

#### القسم 1 — ملخص تنفيذي (Executive Summary)
- الحالة الإجمالية بصيغة واضحة: **Ready / Conditional / Not Ready**
- جدول Blockers الحقيقية المتبقية فقط
- جدول العناصر التي تم التحقق منها فعلياً (مع تاريخ التحقق)
- مصفوفة المخاطر الحالية حسب الأثر/الاحتمال

#### القسم 2 — حالة كل بند سابق (Verification Matrix)
لكل بند من الـ 35 بنداً السابقة، حالة صريحة:
- ✅ **محقق فعلياً** (مع رقم الملف/السطر/الـ migration)
- ⚠️ **محقق جزئياً** (ما المتبقي بالضبط)
- ❌ **لم يُنفذ** (يبقى blocker إن كان حرجاً)
- 🔄 **تغير المسار** (تم حله بطريقة مختلفة)

#### القسم 3 — المنطق المحاسبي (تدقيق عميق)
لكل دورة، فحص فعلي ومعادلات قابلة للتحقق:

**3.1 دورة المبيعات الكاملة**
- Invoice → Posting → COGS → Returns → Settlements → Payments
- معادلة التحقق: `customer.balance = Σ(invoices.net_total) − Σ(returns.net_total) − Σ(allocations) + Σ(refunds)`
- نقاط الفشل المعروفة وسلوكها

**3.2 دورة المشتريات الكاملة**
- نفس المنهج مع WAC و variance account 5108

**3.3 المخزون و WAC**
- معادلة: `Σ(inventory_movements.qty × unit_cost) = balance(account 1104)`
- سيناريوهات WAC: شراء، مرتجع شراء بسعر مختلف، تسوية، مرتجع بيع
- التحقق من symmetry بعد إصلاح `net_total` للمرتجعات

**3.4 ميزان المراجعة**
- شرط القبول: `Σ debits = Σ credits` بدون فروقات تقريب
- اختبار: تشغيل عشوائي لـ N معاملات والتحقق من = 0 بالضبط

**3.5 الإقفال السنوي**
- تحقق من حماية قيود `entry_type = 'closing'`
- ربط الإقفال بـ `locked_until_date` تلقائياً

#### القسم 4 — الأمان والصلاحيات
- جرد فعلي لكل سياسات RLS الحساسة
- مصفوفة صلاحيات admin / accountant / sales
- فحص edge functions الإدارية (تحقق `has_role` صراحة)
- حالة MFA و leaked password protection

#### القسم 5 — الأداء والحدود التشغيلية
- جرد الصفحات: paged ✅ / non-paged ❌ (`Accounts` ما زالت تحتاج)
- الفهارس المطلوبة على الأعمدة عالية الاستعلام
- حجم الـ bundle بعد lazy loading
- استعلامات > 500ms المعروفة

#### القسم 6 — تجربة المستخدم
- حالة ErrorBoundary (موجودة على الجذر ✅)
- حالة `useBeforeUnload` + `navGuard.allowNext()` في كل النماذج
- رسائل الخطأ العربية الموحدة
- طباعة PDF عربي + تصدير Excel/CSV

#### القسم 7 — البيانات والاسترجاع
- خطة backup/restore (RTO/RPO)
- اختبار `database-backup` edge function
- خطة الترحيل من بيانات الاختبار إلى الإنتاج

#### القسم 8 — Blockers النهائية
قائمة قصيرة جداً: فقط ما يجب إغلاقه قبل الإنتاج.

#### القسم 9 — مخاطر معروفة مقبولة
ما يُسمح بالنشر معه + خطة المعالجة اللاحقة.

#### القسم 10 — معايير قبول النشر (Go/No-Go Criteria)
قائمة معايير قابلة للقياس — كلها يجب أن تكون ✅ قبل النشر.

#### القسم 11 — خطة الـ Sprints المتبقية
Sprint 1: Blockers | Sprint 2: High | Sprint 3: Polish — مع تقدير زمني واقعي.

## الملف الثاني — `docs/MANUAL_TEST_PLAN_V3.md`

خطة اختبار يدوي شاملة، **مبنية على المخاطر**، مع أرقام متوقعة دقيقة وقواعد قبول/رفض صريحة.

### هيكل الملف

#### الجزء 1 — الإطار العام
- البيئة المطلوبة (DB نظيف، seed-system)
- جدول الأدوار المستخدمة في الاختبار
- نموذج توقيع لكل مرحلة (المختبِر / التاريخ / النتيجة / الملاحظات)
- مفاتيح القبول: ✅ Pass / ❌ Fail / ⚠️ Pass with notes

#### الجزء 2 — اختبارات Smoke (15 دقيقة)
أساسيات يجب أن تعمل قبل أي اختبار آخر:
- تسجيل دخول + MFA
- تحميل الـ Dashboard
- إنشاء عميل/مورد/منتج
- ترحيل فاتورة بيع بسيطة
- التحقق من ميزان المراجعة = 0

#### الجزء 3 — اختبارات الدورة المحاسبية الكاملة
**كل سيناريو يحتوي:**
- البيانات المُدخلة (دقيقة بالأرقام)
- الأرقام المتوقعة (في كل خانة)
- القيود المتوقعة (جدول مدين/دائن)
- معادلات التحقق
- معايير القبول/الرفض

**السيناريوهات:**
1. شراء → بيع → مرتجع بيع → مرتجع شراء (بدون ضريبة)
2. نفس الدورة بضريبة 15%
3. الخصومات: سطر فقط / إجمالي فقط / منع الجمع
4. WAC مع 3 أسعار شراء مختلفة
5. مرتجع شراء بسعر ≠ WAC (variance → 5108)
6. تسوية المخزون (زيادة/نقص)
7. مدفوعات وتسويات متعددة لفاتورة واحدة
8. مرتجع → تسوية مع فاتورة لاحقة
9. **متوسط سعر البيع** (التحقق من إصلاح net_total الجديد)

#### الجزء 4 — اختبارات التحقق من الأرقام (Numerical Verification)
استعلامات SQL مع نتائج متوقعة:
```sql
-- ميزان المراجعة = 0
SELECT SUM(debit) - SUM(credit) FROM journal_entry_lines 
WHERE journal_entry_id IN (SELECT id FROM journal_entries WHERE status='posted');
-- متوقع: 0.00 بالضبط

-- مطابقة المخزون مع GL
-- مطابقة customer.balance مع كشف الحساب
-- مطابقة paid_amount مع التخصيصات
```

#### الجزء 5 — اختبارات Edge Cases (الحالات الحدية)
- ترحيل فاتورة بتاريخ ضمن فترة مغلقة → يجب أن يُرفض
- مرتجع كمية > الأصلية → يجب أن يُرفض (UI + API)
- ترحيل فاتورتين متوازيتين تستهلكان آخر قطعة → اختبار concurrency
- دفعة بمبلغ صفر/سالب → يجب أن تُرفض
- إلغاء قيد إقفال → يجب أن يُرفض
- محاولة تعديل قيد آلي → يجب أن تُرفض

#### الجزء 6 — اختبارات الأمان والصلاحيات
**مصفوفة:** لكل دور (admin/accountant/sales) × كل صفحة → نتيجة متوقعة (وصول/منع).
- محاولة وصول مباشر للـ URL بدور غير مصرح
- اختبار RLS عبر استعلامات API مباشرة بـ JWT لكل دور
- اختبار MFA enforcement
- audit_log لا يُقرأ إلا للـ admin

#### الجزء 7 — اختبارات التقارير
لكل تقرير:
- المدخلات (الفترة، الفلاتر)
- الأرقام المتوقعة (محسوبة يدوياً من بيانات الاختبار)
- التصدير: PDF عربي صحيح / Excel يفتح بدون فساد / CSV بـ UTF-8 BOM
- المقارنة الفترية (KPI مع الفترة السابقة)

تشمل: ميزان المراجعة، قائمة الدخل، الميزانية، التدفقات النقدية، المبيعات، المشتريات، المخزون، أعمار الديون، الأداء، تحليل المنتجات، 7 تقارير دوران المخزون.

#### الجزء 8 — اختبارات تجربة المستخدم
- **تحذير الخروج بدون حفظ** في كل النماذج (يظهر/لا يظهر بشكل صحيح)
- التنقل + Breadcrumb
- الفلاتر والبحث
- الإدخال السريع (Tab/Enter)
- الموبايل (375px+)
- الطباعة العربية RTL

#### الجزء 9 — اختبارات الأداء
- تحميل صفحة بـ 5000 عميل → استجابة < 2s
- تقرير على 10000 معاملة → < 3s
- bundle size لأول تحميل
- Lighthouse: Performance ≥ 80, Accessibility ≥ 90

#### الجزء 10 — اختبارات الاسترجاع والصيانة
- backup → reset → restore → التحقق من تطابق البيانات
- تشغيل seed-system على DB نظيف
- migration من نسخة أقدم

#### الجزء 11 — اختبار قبول نهائي (UAT Sign-off)
نموذج توقيع نهائي يربط كل الأقسام ويحدد قرار الانتقال للإنتاج.

## منهجية البناء

### قبل الكتابة
1. فحص فعلي لكل ملف ادعت الوثائق السابقة شيئاً عنه (App.tsx, Accounts.tsx, navigation-guard, RLS، إلخ).
2. قراءة آخر migrations للتأكد من حالة triggers, locks, numeric precision.
3. فحص حالة كل blocker سابق: هل تم إصلاحه فعلاً أم لا.

### مبادئ الكتابة
- **لا افتراضات.** كل ادعاء يُسند بمرجع (ملف:سطر أو migration).
- **أرقام دقيقة.** كل سيناريو اختبار يعطي القيمة المتوقعة بالضبط (ليس "تقريباً").
- **عربي واضح + جداول.** سهل القراءة لغير التقني.
- **قابل للتنفيذ.** كل خطوة يمكن لشخص جديد تنفيذها بدون سؤال.

### الناتج النهائي
- ملفان مستقلان قابلان للطباعة كـ PDF أو مشاركتهما مع طرف ثالث للمراجعة.
- مرجع وحيد لقرار "Go / No-Go" للإنتاج.
- أساس لكل sprint قادم بدون تكرار العمل.

## ما لن يتم في هذه الخطة
- **لا تعديلات على الكود أو الـ DB.** فقط ملفات توثيق.
- لا إصلاحات للـ blockers — هذه الخطة تُحدّدها وتُوثّقها فقط، ثم نتفق على sprint منفصل لإصلاحها.
- لا حذف للملفات القديمة (`PRODUCTION_READINESS_AUDIT.md`, `MANUAL_TEST_PLAN.md`) — تبقى للمرجعية التاريخية، والملفان الجديدان `_V3` يحلان محلهما عملياً.
